# GitHub Repo Research 产品定义

## 一句话价值

给我一个 GitHub 仓库链接，我在一轮内告诉你这个项目值不值得继续看，核心代码在哪，依赖什么，适不适合拿来复用。

## 目标用户

- 开发者：快速理解陌生仓库，减少踩坑时间。
- 独立黑客：找竞品、找方案、找可复用实现。
- 技术运营或 PM：判断一个开源项目是否值得接入。
- 投研或顾问：对技术项目做轻量尽调。
- AI agent 使用者：让 agent 在开始深入前先做仓库级判断。

## 核心场景

- 看一个新仓库前先判断是否值得投入 30 分钟以上。
- 找相似产品或竞品仓库，先做快速筛选。
- 看到热门 repo，不知道真正价值在哪。
- 想找 API、技术栈、关键入口文件、成熟度信号。

## 为什么这个 skill 容易落地

- 搜索意图明确，用户本来就会搜 GitHub、repo、analyzer、summary 这类词。
- 输入门槛低，只要一个链接。
- 输出容易感知价值，用户很快就能判断“有用/没用”。
- 后续变体自然，容易形成 skill factory。

## 最小可用版本

- 输入：1 个 GitHub repo URL。
- 输出：
  - 项目是干什么的
  - 适合谁
  - 技术栈
  - 关键入口文件
  - 外部 API / 服务依赖
  - 成熟度 / 风险信号
  - 是否值得继续研究

## 付费钩子

- 免费版：只看 1 个仓库，只做浅层分析。
- 专业版：支持更深目录遍历、关键文件抽取、Issue/PR 风险扫描、依赖风险扫描。
- 团队版：支持多仓库比较、共享模版、watchlist、变更提醒。

## 推荐标题

- GitHub Repo Research
- GitHub Repo Analyzer
- GitHub Repo Summary
- GitHub Repo Due Diligence

## 不要做成什么样

- 不要一上来就讲 agent 架构和复杂工作流。
- 不要要求用户先配置一堆参数。
- 不要输出一大段泛泛而谈的 README 复述。
- 不要把第一次输出做成代码审计报告，太重。
